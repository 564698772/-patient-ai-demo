# Product Requirements Document - Patient Medical AI System

**作者：** Dd
**日期：** 2026-04-06
**项目类型：** Web 应用（Vue SPA + Spring Boot REST API）| **领域：** 医疗健康 | **复杂度：** 高 | **背景：** 全新项目

---

## Executive Summary

患者就医 AI 推荐系统是一款面向普通患者的 C 端 Web 应用，解决患者在突发或紧急就医场景下"不知道去哪、不知道能不能看到"的核心痛点。系统通过通义千问 AI 接口理解患者输入的自然语言症状描述，自动匹配对应科室，结合高德地图 API 定位患者当前位置，推荐附近适合就诊的医院列表。

目标用户为有即时就医需求的普通患者，尤其覆盖外地人、老年人、急症患者等在陌生环境中缺乏就医经验的群体。

### 核心差异化

现有替代方案（百度搜索、高德地图搜"医院"）只能给出地点列表，无法回答"我的症状该看哪科"。本系统的差异化在于：

- **症状 → 科室映射**：AI 自动分析症状并推荐匹配科室，用户无需具备医学知识
- **位置 + 可就诊性结合**：不只是"附近有医院"，而是"附近能看你这个病的医院"
- **急症优先机制**：识别高危症状，直接提示拨打 120，患者安全优先于推荐流程
- **时间导向**：MVP 展示医院基础信息与距离，后续迭代接入号源数据

**一句话价值主张：描述你的症状，我来告诉你附近哪家医院现在就能看你。**

## Success Criteria

### User Success

- 患者从打开页面到看到医院推荐结果，**全程不超过 30 秒**（含症状输入 + AI 分析 + 地图返回）
- AI 推荐结果包含**科室名称 + 推荐理由**，患者无需二次搜索即可判断是否合适
- 推荐列表展示**至少 3 家**附近医院，含距离、地址、擅长科室
- 用户无需注册登录即可使用核心功能

### Business Success

- 完整实现 MVP 功能，可作为技术作品集展示
- 前后端代码结构清晰，符合 Spring Boot + Vue 最佳实践，具备可扩展性
- 系统可本地运行并演示完整就医查询流程

### Technical Success

- AI 症状分析（通义千问）响应时间 **< 3 秒**；后端超时上限 5 秒，超时后返回兜底文案
- 高德地图周边搜索返回 **≥ 3 家**医院 POI 结果
- 前后端接口稳定，无崩溃性错误
- 患者位置数据**不持久化存储**（隐私合规基线）

### Measurable Outcomes

| 指标 | 目标值 |
|------|--------|
| 端到端查询响应时间 | ≤ 30 秒 |
| AI 接口响应时间 | < 3 秒 |
| 推荐医院数量 | ≥ 3 家 |
| 页面首屏加载（FCP） | < 2 秒 |
| 用户操作步骤 | ≤ 3 步完成查询 |
| 急症词识别召回率 | ≥ 95% |

## User Journeys

### 旅程一：急症患者 — 核心成功路径

**用户：小李，28岁，在外出差**

小李在酒店突然感到腹部剧烈疼痛，人生地不熟，不知道附近哪家医院能看急腹症。

- **开场**：手机浏览器打开页面，系统请求位置权限，小李点击"允许"。
- **行动**：输入"肚子右下方很痛，已经两小时了，越来越严重"，点击查询。
- **高潮**：3秒内，AI 返回"症状提示可能为急腹症（阑尾炎可疑），建议尽快前往外科或急诊科就诊"，同时展示附近3家医院，距离最近的"800米外，设有急诊科"排在最前。
- **结局**：小李叫车直奔最近医院急诊，全程不到1分钟。

*揭示的能力：症状输入、AI 分析、GPS 定位、医院列表、距离排序、科室标注*

### 旅程二：慢性患者 — 常规就医路径

**用户：王阿姨，62岁，本地居民**

王阿姨最近总是头晕，想去看医生，但不知道挂什么科。

- **开场**：子女帮她打开页面，系统自动定位。
- **行动**：输入"最近总是头晕，站起来就转，有时候耳鸣"。
- **高潮**：AI 返回"可能与耳石症或血压波动相关，建议就诊耳鼻喉科或内科/神经科"，列出附近5家医院并标注推荐科室。
- **结局**：王阿姨选择最近的二甲医院，知道挂"耳鼻喉科"，不再茫然。

*揭示的能力：多科室推荐、医院等级展示、步行距离*

### 旅程三：GPS 失效 — 边缘场景

**用户：张先生，在地下商场**

张先生感到胸闷，手机 GPS 信号弱，自动定位失败。

- **开场**：页面提示"无法获取位置，请手动输入地址"。
- **行动**：输入"北京市朝阳区建国路"作为参考位置。
- **高潮**：系统以该地址为中心执行周边搜索，正常返回医院列表。
- **结局**：成功找到附近医院，Fallback 处理流畅，无崩溃。

*揭示的能力：手动地址输入、Fallback 定位逻辑、友好错误提示*

### 旅程四：系统管理员 — 运维路径

**用户：开发者/运维人员**

- **开场**：通过 Spring Boot Actuator 检查系统健康状态。
- **行动**：查看 API 调用日志，确认通义千问与高德接口响应正常，检查 API 调用量。
- **高潮**：发现某次查询因 AI 超时返回了兜底提示，记录并排查。
- **结局**：系统稳定运行，异常有日志可查。

*揭示的能力：接口调用日志、兜底响应机制、健康检查接口*

| 旅程 | 揭示的核心能力 |
|------|--------------|
| 急症患者 | 症状输入、AI 分析、GPS 定位、医院列表、距离排序 |
| 慢性患者 | 多科室推荐、医院等级、步行距离 |
| GPS 失效 | 手动地址输入、Fallback 逻辑、友好错误提示 |

## Domain-Specific Requirements

### 监管定位声明

本产品定位为**导诊工具**（非互联网诊疗）：仅提供就医方向建议，不提供诊断、治疗方案或处方，不建立医患关系。依据国家卫生健康委员会《互联网诊疗管理办法（试行）》（2018）及相关细则，本产品不属于互联网医疗监管范畴，无需取得互联网医院执照。AI 输出内容责任边界：系统提供参考建议，最终就医决策由用户自主做出；平台不承担医疗诊断责任。

### Compliance & Regulatory（合规与法规）

- **《个人信息保护法》（PIPL）基线**：位置信息属于敏感个人信息，需明确告知用途，获取明确授权后使用，不得超范围收集
- **医疗 AI 免责声明**（必须）：页面显著位置展示"本系统建议仅供参考，不构成医疗诊断，请遵医嘱"
- **数据最小化原则**：仅采集症状文本 + 位置坐标，不采集姓名、身份证、手机号等个人身份信息
- **不持久化患者数据**：查询结果不写入数据库，会话结束后数据不保留

### Technical Constraints（技术约束）

- **位置数据不上传服务器**：地理坐标仅在前端处理后传给高德 API，后端不存储用户坐标
- **HTTPS 传输**：生产环境所有接口必须走 HTTPS
- **API Key 安全**：通义千问 Key、高德地图 Key 存放于后端环境变量，不暴露在前端代码或版本控制中
- **AI 输出内容过滤**：通义千问返回内容需做基础过滤，避免输出极端或误导性医疗建议

### Emergency Detection（急症优先机制）— MVP 必须实现

当 AI 分析识别到高危信号词（胸痛、呼吸困难、意识丧失、大量出血等）时，系统**优先展示急救提示**而非常规医院推荐：

> "您描述的症状可能属于紧急情况，请立即拨打急救电话 **120**，或让他人送您前往最近医院急诊。"

此机制在后端 AI 分析阶段判断，前端根据响应标志位（`isEmergency: true`）渲染对应 UI。

### Risk Mitigations（风险缓解）

| 风险 | 缓解措施 |
|------|---------|
| AI 给出错误医疗建议 | 强制免责声明 + 展示推荐理由，用户自主判断 |
| 通义千问 API 超时/不可用 | 后端 5 秒超时，返回兜底文案"AI 分析暂时不可用，请直接拨打 120 或前往最近医院" |
| 高德 API 免费额度耗尽 | 后端记录调用次数，超限时返回友好提示 |
| 用户位置获取失败 | 提供手动输入地址的 Fallback，附操作引导文案 |
| 患者延误就医 | 急症关键词检测，直接提示"请立即拨打 120" |

## Innovation & Novel Patterns

### Detected Innovation Areas

本产品的核心创新在于**语言鸿沟桥接**：利用大语言模型将患者口语化症状描述转化为结构化医疗科室推荐，再与地理位置服务结合，形成"症状 → 科室 → 附近医院"的完整推荐链路。

| 创新点 | 说明 |
|--------|------|
| 语言桥接 | 患者描述"肚子右下方痛" → 系统映射为"外科/急诊科"，消除普通人与医院科室体系间的信息鸿沟 |
| 急症升级机制 | 根据症状危险程度动态切换响应模式：常规推荐 vs 拨打 120 |

### Market Context

| 现有产品类型 | 代表产品 | 能力缺口 |
|------------|---------|---------|
| 地图类 | 高德/百度地图 | 提供地点，不理解症状 |
| 问诊类 | 春雨医生、丁香医生 | 在线问诊，非就近推荐，需注册 |
| 挂号类 | 京医通、健康160 | 需要知道科室才能操作 |

**本产品填补的空白**：无需注册、无需知道科室、基于当前位置、30 秒内完成的轻量级导诊工具。

### Validation Approach

- AI 对 10 个常见症状的科室推荐准确率 ≥ 80%
- 给定位置坐标，稳定返回 ≥ 3 家医院结果
- 高危词急症识别召回率 ≥ 95%

## Web Application Specific Requirements

### Architecture Overview

**Vue.js SPA + Spring Boot REST API，前后端分离。** 前端负责交互与渲染，后端负责 AI 代理调用与地图数据聚合，面向移动端患者用户，Mobile First 设计优先。

### Browser Matrix（浏览器支持）

| 浏览器 | 支持级别 |
|--------|---------|
| Chrome（Android/Desktop） | 完全支持 |
| Safari（iOS） | 完全支持 |
| 微信内置浏览器（X5） | 完全支持 |
| Firefox | 基本支持 |
| IE 11 及以下 | 不支持 |

### SEO Strategy

本产品为即时导诊工具，用户通过直接链接、二维码或口碑分享访问，**SEO 不在 MVP 及后续版本范围内**。页面不需要针对搜索引擎优化，无需 sitemap 或结构化数据标记。

### Responsive Design（响应式设计）

- Mobile First，以 375px（iPhone SE）为基准断点，向上适配平板和桌面
- 核心交互区（输入框、查询按钮、结果列表）在 375px 下完整可用，无需横向滚动
- 医院结果卡片：单列布局（移动端）/ 两列布局（768px+）

### Implementation Stack（实现要点）

| 技术 | 用途 |
|------|------|
| Vue Router | 首页（症状输入）→ 结果页（医院列表）单页路由 |
| Axios | 前端 HTTP 客户端，调用 Spring Boot 后端 |
| Pinia | 轻量状态管理，管理查询状态与结果数据 |
| Vite | 构建工具，热更新与生产打包优化 |
| 高德地图 JS SDK | 可选地图视图渲染（MVP 可先用列表替代） |

## Project Scoping & Phased Development

### MVP Strategy

**方式：** 问题解决型 MVP — 验证"症状 → 科室 → 附近医院"核心链路能跑通、能帮到人，再迭代。
**资源：** 1名全栈开发者，预计 2-4 周完成。

### Phase 1 — MVP 必须具备的能力

| # | 功能 | 说明 |
|---|------|------|
| 1 | 症状文本输入 | 自然语言输入框，附 placeholder 示例 |
| 2 | AI 症状分析 | 后端调用通义千问，返回科室建议 + 推荐理由 |
| 3 | 急症检测 | AI 响应含 `isEmergency` 标志位，前端显示 120 提示 |
| 4 | GPS 自动定位 | 浏览器 Geolocation API |
| 5 | 手动地址输入 | GPS 失败时 Fallback |
| 6 | 附近医院查询 | 高德地图周边搜索 API，后端调用 |
| 7 | 医院列表展示 | 名称、距离、地址、推荐科室，按距离排序 |
| 8 | 免责声明 | 页面固定展示 |
| 9 | AI 超时兜底 | 5 秒超时后返回默认提示文案 |

- 接入第三方号源 API，显示实时挂号余量
- 接入医保数据，筛选医保可用医院

### Risk Mitigation Strategy

| 风险类型 | 风险 | 缓解措施 |

## Functional Requirements

### 症状输入与 AI 分析

- **FR1**：患者可通过自然语言文本输入描述自身症状
- **FR2**：系统可将患者的症状描述提交至 AI 服务进行分析
- **FR3**：系统可基于 AI 分析结果向患者展示推荐就诊科室及简要说明
- **FR4**：系统可识别症状描述中的高危紧急信号（如胸痛、呼吸困难、意识丧失、大量出血），并隐藏常规医院推荐列表、置顶展示急救提示横幅
- **FR5**：当急症信号被识别时，系统可向患者展示急救提示（含 120 拨打建议）
- **FR6**：当 AI 分析 5 秒内无响应时，系统向患者展示固定兜底提示："AI 分析暂时不可用，请直接拨打 120 或前往最近医院"

### 位置获取与定位

- **FR7**：患者可授权系统获取其当前地理位置
- **FR8**：患者可在自动定位失败时手动输入地址作为替代位置
- **FR9**：系统可基于患者位置（自动或手动）执行附近医院查询

### 医院查询与推荐

- **FR10**：系统可查询患者当前位置周边的医院列表
- **FR11**：系统可将医院列表按与患者的距离由近至远排序展示
- **FR12**：系统可在每家医院条目中展示医院名称、距离、地址信息
- **FR13**：系统可在医院条目中标注与 AI 推荐科室相匹配的科室信息
- **FR14**：系统可保证推荐列表至少包含 3 家医院结果（数据可用时）

### 结果展示与交互

- **FR15**：患者可查看本次查询的完整结果页面（AI 分析 + 医院列表）
- **FR16**：患者可在结果页面看到 AI 推荐理由，以便自主判断是否合适
- **FR17**：患者可获取每家医院的地图导航链接
- **FR18**：系统在每个页面固定展示医疗 AI 免责声明，声明文字字号 ≥ 14px

### 合规与安全

- **FR19**：系统可在首次获取位置前向患者明确告知位置数据的使用目的
- **FR20**：系统可确保患者症状与位置数据不被持久化存储至服务器

### 系统运维与可观测性

- **FR22**：运维人员可通过健康检查接口确认系统服务是否正常运行
- **FR23**：系统可记录 AI 服务与地图服务的接口调用日志，供排查使用
- **FR24**：系统在外部服务异常时向患者展示友好错误提示，结果页面保持可见且无空白屏

## Non-Functional Requirements

### Performance（性能）

| 指标 | 要求 | 说明 |

| 首屏内容渲染（FCP） | < 2 秒 | 4G 网络条件下，前端资源总体积 < 500KB（gzip 后） |

### Security（安全）

- 所有外部服务 API 密钥存储于服务端环境变量，不得出现在客户端代码或版本控制中
- 生产环境全站强制 HTTPS，HTTP 请求重定向至 HTTPS
- 传递至服务端的数据仅限症状文本与位置坐标，服务端不记录、不持久化
- 服务端对跨域请求（CORS）做严格配置，仅允许指定前端域名访问
- AI 返回内容在服务端做基础过滤，拦截明显的注入或敏感信息

### Reliability（可靠性）

- 任意单个外部依赖（通义千问 / 高德地图）故障不导致系统整体崩溃，均有独立兜底响应
- 后端接口异常返回结构化 JSON 错误响应，HTTP 状态码语义正确（4xx / 5xx）
- 前端对所有异步请求做错误边界处理，用户不会看到空白页或未处理的 JS 报错
- Spring Boot Actuator `/health` 端点可用，响应时间 < 500ms

### Accessibility（无障碍访问）

- 正文字体最小 16px，主操作区文字 ≥ 18px
- 颜色对比度符合 WCAG 2.1 AA（正文 ≥ 4.5:1，大字 ≥ 3:1）
- 急症警告不依赖颜色单一传达信息，同时使用图标与文字
- 所有表单输入框具备明确 label 或 aria-label
- 触控目标最小尺寸 44×44px

### Integration（集成）

- **通义千问 DashScope API**：后端代理调用，超时 5 秒，失败率 > 3 次/分钟时记录告警日志；前端不直接访问
- **高德地图周边搜索 API**：后端调用，半径默认 5km，可配置；返回结果按距离升序
- **高德地图 JS SDK**（前端）：仅用于可选的地图视图渲染，不影响核心列表功能
- 前后端 API 通信格式统一为 JSON，字符集 UTF-8
